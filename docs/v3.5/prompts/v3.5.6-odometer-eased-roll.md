# v3.5.6 Odometer Movement

Implement the v3.5.6 slice only.

This file is one v3.5 implementation slice. The filename is legacy; this slice now defines the odometer main movement phase.

If invoked from a broad instruction such as "implement the next slice", first confirm this file matches the first unchecked slice in `docs/v3.5/ImplementationState.md`.

Do not implement this slice if an earlier slice is still unchecked.

## Goal

Use `odometer.movement` as the single source of truth for odometer wheel movement.

Do not require or use `realism.movement_policy` for odometer movement.

## Movement values

Supported odometer movement values:

- `instant`: digit display jumps immediately to the target value. No animation.
- `linear`: wheel rolls from old digit position to target digit position at constant speed.
- `ease_out`: wheel starts moving fast, then slows into the target.
- `bell`: wheel starts slow, speeds up through the middle, then slows into the target. Bell-curve velocity / smoothstep-style movement.
- `smooth`: recognised but not implemented. Emit a clear warning and fall back to `instant`.
- `click`: recognised but not implemented. Emit a clear warning and fall back to `instant`.

Unknown odometer movement values must fail configuration loading with a clear error.

## Required behaviour

When an odometer value changes and movement is animated:

```text
previous digit position -> movement curve -> target digit position
```

At completion, the odometer must settle exactly on the target value.

Do not implement movement by permanently feeding fractional numeric odometer values back into the source display value.

Bad model:

```text
displayValue = 5998.5
```

Good model:

```text
wheel position = movement phase from previous digit position to target digit position
```

## Phase model

Structure the odometer movement code so future effects can compose around it:

```text
pre-movement modifiers -> main movement -> post-movement modifiers -> static offsets
```

For this slice:

- pre-movement modifiers are no-op;
- post-movement modifiers are no-op;
- static offsets such as existing `drum_slop` may still apply;
- main movement is controlled by `odometer.movement`.

Do not implement `carry_drag`, `snap_settle`, or `backlash` in this slice.

## Fixtures and docs

Update odometer preview YAMLs and examples to use:

```yaml
odometer:
  movement: bell
```

where the intended visual showcase is the bell movement.

Remove `realism.movement_policy` from odometer fixtures/examples.

If an old fixture still uses `smooth` or `click`, it should exercise the warning/fallback behaviour intentionally. Otherwise replace it.

## Tests

Add or update tests covering:

- `movement: instant` jumps directly to the target.
- `movement: linear` animates over ticks and settles exactly on target.
- `movement: ease_out` animates over ticks and settles exactly on target.
- `movement: bell` animates over ticks and settles exactly on target.
- `movement: smooth` emits a warning and behaves as `instant`.
- `movement: click` emits a warning and behaves as `instant`.
- unknown movement values fail configuration loading.
- odometer movement does not require `realism.movement_policy`.
- the completed movement does not retain fractional eased-roll offset as settled state.

Useful odometer value examples:

- `5999 -> 5998` settles exactly on `5998`.
- `5998 -> 5997` settles exactly on `5997`.
- `5990 -> 5989` settles exactly on `5989`.
