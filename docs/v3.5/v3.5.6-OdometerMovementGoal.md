# v3.5.6 Odometer Movement Goal

## Goal

v3.5.6 defines the **main odometer wheel movement phase**.

`movement` is the single movement knob. It should remain acceptable across gauge types for now, even where non-odometer gauge types initially stub or keep immediate behaviour.

For odometers, it should use one odometer-specific field:

```yaml
odometer:
  movement: instant | linear | ease_out | bell | smooth | click
```

Do not use or recommend `realism.movement_policy` for odometer movement.

## Movement values

| Value | Status | Behaviour |
|---|---|---|
| `instant` | implemented | Digit changes immediately. No animation. |
| `linear` | implemented | Wheel rolls from old digit to target digit at constant speed. |
| `ease_out` | implemented | Wheel starts fast, then slows into the target. |
| `bell` | implemented | Wheel starts slow, speeds up through the middle, then slows into the target. Bell-curve velocity. |
| `smooth` | recognised only | Warn that this is reserved for future enhancement; fall back to `instant`. |
| `click` | recognised only | Warn that this is reserved for future stepped-click enhancement; fall back to `instant`. |

## Phase model

Odometer realism should be composable:

```text
pre-movement modifiers -> main movement -> post-movement modifiers -> static offsets
```

For v3.5.6:

```text
none -> instant / linear / ease_out / bell -> none -> existing static offsets
```

Future slices fit around this:

| Slice | Feature | Phase |
|---|---|---|
| v3.5.2 | `wraparound` | movement path rule |
| v3.5.3 | `drum_slop` | static resting offset |
| v3.5.6 | `movement` | main movement phase |
| v3.5.7 | `carry_drag` / 9-drag | pre/overlap movement on neighbouring wheels |
| v3.5.14 | `snap_settle` | post-movement tail |
| v3.5.15 | `backlash` | pre/post direction-change slack |

## Implementation rule

v3.5.6 must not render by permanently feeding fractional numeric odometer values back into the source display value.

Bad model:

```text
displayValue = 5998.5
```

Good model:

```text
from digit position -> movement phase -> target digit position
```

At the end of the movement phase, the handover position must be exactly the target digit position. Later post-movement effects such as `snap_settle` may start from that handover position.

## Scope

v3.5.6 must not implement:

- `carry_drag`
- `snap_settle`
- `backlash`
- future stepped `click`
- future generic `smooth`
- general physics
- idle animation
- asset-root loading changes

v3.5.6 only defines the main wheel movement phase.
