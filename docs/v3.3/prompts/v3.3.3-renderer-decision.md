# v3.3.3 - renderer decision

## Goal

Use the v3.3.2 comparison results to decide whether Ebiten should continue, become primary later, pause, or be abandoned.

## Required context

Read:

- `docs/v3.3/ReleasePlan.md`
- `docs/v3.3/ImplementationState.md`
- `docs/v3.3/BaselineDashboardVerification.md`
- the recorded v3.3.2 comparison results

## Scope

- Review Fyne vs Ebiten measurements.
- Compare render stats, superseded frames, render duration, CPU use, smoothness, reliability, and code complexity.
- Record one clear decision.
- Identify the next slice only if justified.

## Decision options

| Outcome | Meaning |
|---|---|
| Continue Ebiten | Ebiten is clearly useful but not ready to become primary. |
| Promote Ebiten path | Ebiten is clearly faster/smoother and worth making primary in a later slice. |
| Keep Fyne primary | Ebiten is not enough better to justify churn. |
| Pause/abandon Ebiten | Ebiten is unstable, awkward, or Pi-hostile. |

## Decision guidance

- A 10-20% Ebiten win is probably not worth switching.
- A 30-80% win may be worth further work if the code remains simple.
- A 2x+ win plus smoother Pi behaviour is a strong continue signal.
- Fyne remains primary unless Ebiten clearly wins.

## Acceptance criteria

- Decision is documented with supporting evidence.
- Default renderer state is explicit.
- Next work is identified only if the decision needs it.
- No vibe-only decision is accepted. Vibes may attend the meeting, but they do not get a vote without numbers.

## Do not

- Do not continue Ebiten out of sunk-cost stubbornness.
- Do not abandon Fyne unless Ebiten proves itself.
- Do not start another renderer without a fresh plan.
