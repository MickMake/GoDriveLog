# v3.3.1 - Ebiten renderer spike

## Goal

Add an experimental Ebiten renderer backend beside Fyne and test it through the real GoDriveLog dashboard path.

## Required context

Read:

- `docs/v3.3/ReleasePlan.md`
- `docs/v3.3/ImplementationState.md`
- `docs/v3.3/BaselineDashboardVerification.md`
- `examples/baseline-dashboard.yaml`

## Scope

- Add `--renderer fyne|ebiten`.
- Keep Fyne as the default.
- Add a narrow Ebiten adapter package.
- Reuse the existing v3 dashboard scene model.
- Support the baseline dashboard enough for comparison.
- Start with Ebiten runtime radial needle rotation.
- Document prepared/cached radial needle frames as the fallback if runtime rotation is too expensive.

## Full-path requirement

Ebiten must use:

```text
OBD / harness source
-> prepared vehicle/sensor data
-> runtime event path
-> dashboard scene generation
-> display sink / latest submission
-> selected renderer adapter
-> screen
```

Do not build a renderer-private demo path. If the benchmark bypasses the real path, it is not evidence; it is theatre with pixels.

## Acceptance criteria

- `--renderer fyne` still works.
- `--renderer ebiten` runs the baseline dashboard path.
- Fyne remains default.
- The same config, vehicle, sensors, runtime events, and scene model feed both renderers.
- Ebiten renders static layers, seven-segment digits, minus glyph, numeric RPM, and radial RPM sufficiently for comparison.
- Comparable display stats are still emitted.
- No gauge package redesign is introduced.

## Do not

- Do not remove Fyne.
- Do not promote Ebiten yet.
- Do not add sensor overrides.
- Do not add clusters, inheritance, or procedural artwork.
- Do not chase polish before measurement.
