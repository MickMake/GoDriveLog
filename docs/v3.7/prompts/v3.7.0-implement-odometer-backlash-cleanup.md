# v3.7.0 Audit Odometer Backlash

Run the v3.7.0 audit slice only.

This file is an audit prompt, not an implementation prompt.

Do not implement odometer `backlash` in this slice. The job is to determine the current repository state and decide whether a later implementation slice is needed.

## Goal

Audit odometer `backlash` support across docs, config parsing, validation, runtime behaviour, tests, and preview fixtures.

The audit must answer whether `backlash` is:

- already implemented;
- partially implemented;
- missing;
- wrongly documented;
- or intentionally deferred.

## Background

Earlier documentation lists `backlash` as an odometer realism feature. Before changing code, verify the current repository state:

- parser field support;
- validator support;
- runtime behaviour;
- tests;
- preview fixtures;
- existing docs and release state.

Do not rely only on historical checklist wording. The checklist may be a signpost, or it may be a small bureaucratic poltergeist.

Existing behaviours are not the same thing:

| Existing option | Why it is not backlash |
| --- | --- |
| `drum_slop` | Static wheel alignment imperfection; does not care about direction changes. |
| `carry_drag` | Rollover coupling between wheels; not reverse-direction slack. |
| `snap_settle` | Landing/settle effect; not slack when reversing. |
| `movement: linear`, `ease_out`, `bell` | Movement curves; not mechanical play. |
| `wraparound` | Route choice across digit boundaries; not slack. |

## Audit requirements

Inspect the current repository and report findings for:

1. Documentation references
   - Is `backlash` listed in v3.5 or later docs?
   - Is it marked complete anywhere?
   - Are there contradictions between docs and code?

2. Config parsing
   - Does the config model include `realism.backlash`?
   - Is it boolean-only or structured?
   - Is it accepted in YAML examples or tests?

3. Validation
   - Is `backlash` restricted to odometer gauges?
   - Is it rejected for radial, bar, numeric, segmented, or indicator gauges?

4. Runtime behaviour
   - Is there code that detects odometer direction reversal?
   - Is there a bounded slack/lag effect before following the new direction?
   - Does the wheel settle exactly at the target?

5. Tests
   - Is parsing tested?
   - Is validation tested?
   - Is same-direction movement tested?
   - Is reversed-direction behaviour tested?
   - Is final settling tested?

6. Preview / examples
   - Is there a fixture or documented example showing a reversing odometer with backlash enabled?

## Candidate implementation semantics for later

If a later slice promotes implementation, `backlash` should mean:

> when an odometer-style value reverses direction, wheel movement shows a small bounded amount of mechanical slack before following the new direction and settling exactly on the target.

A later implementation must be:

- odometer-only;
- display-only;
- deterministic;
- bounded and subtle;
- active only when configured;
- safe for forward and reverse value changes;
- settled exactly at the target when movement completes.

It must not mutate source values, logs, exports, configured ranges, or input data.

## Do not

- Do not implement `backlash` in v3.7.0.
- Do not make `backlash` apply to radial, bar, indicator, numeric, or segmented gauges.
- Do not add randomness.
- Do not create a broad physics engine.
- Do not implement `smooth` or `click` as new movement behaviours in this slice.
- Do not change source values or persisted data.
- Do not bundle unrelated backlog items into this slice.
- Do not make this part of v3.6.
