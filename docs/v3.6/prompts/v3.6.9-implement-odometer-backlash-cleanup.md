# v3.6.9 Implement Odometer Backlash Cleanup

Implement the v3.6.9 slice only.

This file is one v3.6 implementation slice.

Do not implement this slice if v3.6.8 is still unchecked.

## Goal

Implement missing odometer `backlash` support and clean up the reserved odometer movement-mode decisions.

`backlash` is required because existing odometer realism can create general mechanical feel, but cannot fully create direction-change slack.

## Background

The v3.5 documentation lists `backlash` as an odometer realism feature. Repository review found documentation/planning references, but no parser field, validator support, runtime behaviour, tests, or preview fixture for actual `backlash` support.

Existing behaviours are not the same thing:

| Existing option | Why it is not backlash |
| --- | --- |
| `drum_slop` | Static wheel alignment imperfection; does not care about direction changes. |
| `carry_drag` | Rollover coupling between wheels; not reverse-direction slack. |
| `snap_settle` | Landing/settle effect; not slack when reversing. |
| `movement: linear`, `ease_out`, `bell` | Movement curves; not mechanical play. |
| `wraparound` | Route choice across digit boundaries; not slack. |

## Required config

Add odometer-only support for:

```yaml
realism:
  backlash: true
```

Keep the first implementation boolean-only unless a small amount/timing shape is already clearly supported by existing parser conventions.

Do not add hidden default behaviour. Existing gauges without `backlash` must render as they did before.

## Required behaviour

`backlash` means:

> when an odometer-style value reverses direction, wheel movement shows a small bounded amount of mechanical slack before following the new direction and settling exactly on the target.

The implementation must be:

- odometer-only;
- display-only;
- deterministic;
- bounded and subtle;
- active only when configured;
- safe for forward and reverse value changes;
- settled exactly at the target when movement completes.

It must not mutate source values, logs, exports, configured ranges, or input data.

## Implementation requirements

- Add config parsing support for `realism.backlash`.
- Validate that `backlash` is only accepted for odometer gauges.
- Add runtime behaviour for direction-change slack during odometer wheel movement.
- Preserve existing behaviour when `backlash` is absent or false.
- Ensure same-direction odometer movement is not visibly affected by `backlash`.
- Ensure direction reversal produces a small slack/lag before the wheel follows the new direction.
- Ensure final wheel offsets settle exactly on the target offsets.
- Keep the behaviour deterministic; do not introduce random jitter.

## Tests

Add or update tests for:

- YAML parsing accepts `realism.backlash: true` for odometers.
- YAML validation rejects `backlash` for non-odometer gauge types.
- Existing odometer movement remains unchanged when `backlash` is absent.
- Same-direction odometer movement is unaffected or effectively equivalent when backlash is enabled.
- Reversed-direction odometer movement applies bounded slack and then catches up.
- Final offsets still settle exactly on target offsets.

## Preview / docs

Add a small preview fixture or documented example that demonstrates a reversing odometer value with `backlash` enabled.

Update relevant v3.5/v3.6 docs so `backlash` is no longer marked as implemented without code.

## Movement-mode cleanup

While doing this slice, document but do not implement these reserved movement modes as new behaviours:

### `smooth`

Do not implement `movement: smooth` as a separate odometer movement mode.

The existing smooth odometer movement modes are:

- `movement: linear`;
- `movement: ease_out`;
- `movement: bell`.

If `smooth` remains accepted by config, keep it explicitly documented as reserved/unsupported or map it deliberately only if the docs and tests say so.

### `click`

Do not implement `movement: click` as a separate odometer movement mode unless a later slice defines distinct stepped-wheel behaviour.

Most click-like mechanical feel should come from:

- `movement: instant`;
- `drum_slop`;
- `carry_drag`;
- `snap_settle`;
- `backlash`.

## Do not

- Do not make `backlash` apply to radial, bar, indicator, numeric, or segmented gauges.
- Do not add randomness.
- Do not create a broad physics engine.
- Do not implement `smooth` or `click` as new movement behaviours in this slice.
- Do not change source values or persisted data.
- Do not bundle unrelated v3.6 backlog items into this slice.
