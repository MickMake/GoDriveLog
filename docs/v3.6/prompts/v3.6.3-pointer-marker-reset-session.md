## Goal

Define and implement reset/session behaviour for pointer marker state.

Pointer marker state is runtime/session state in v3.6. It should be deterministic and easy to reason about, not an accidental persistence layer hiding in the rafters.

## Requirements

- Preserve radial max/min marker behaviour from v3.6.1 and v3.6.2.
- Define marker reset behaviour in code and docs.
- Reset marker state when a gauge instance is created.
- Reset marker state when the dashboard/runtime reloads.
- Reset marker state when relevant gauge config changes.
- Reset marker state when the data source identity changes.
- Ensure disabled markers do not retain stale visible state when re-rendered.
- Ensure enabling a marker starts from the current rendered indicator position or the next observed rendered position, whichever matches the existing render/update architecture most cleanly.
- Keep reset behaviour display-only; do not mutate source values, logs, exports, configured ranges, or input data.
- Add tests for initial state, config-change reset, source-identity reset, disable/enable behaviour, and no persistence across runtime/session recreation.
- Update docs if the implementation needs to choose between current-position initialisation and next-sample initialisation.

## Do not

- Do not add database persistence.
- Do not add manual UI reset controls unless such controls already exist and require only a minimal hook.
- Do not implement bar markers in this slice.
- Do not implement damped secondary markers in this slice.
- Do not add mathematical average.
