# v3.1.4 JSONL daily rotation

Instruction to slice chat: implement only v3.1.4.

v3.1.4

## Goal

Keep JSONL logging simple by using daily log rotation.

## Decision

v3.1 JSONL logging uses daily rotation.

The implementation should make the daily file boundary explicit and documented rather than silently inheriting old logger behaviour.

## Expected behaviour

- Keep JSONL logging as a subscriber to sensor events.
- Write JSONL logs to a daily rotated file.
- Use a predictable date-based file naming convention.
- Create or switch to the next daily file when the local date changes.
- Keep existing log records line-delimited JSON.
- Keep the behaviour simple; do not add multiple rotation modes unless a later slice explicitly requires it.

## OpenDecisions.md

Update this file because this slice closes the JSONL rotation decision as daily rotation.

## Do not

- Add configurable rotation modes in this slice.
- Add a separate logging architecture.
- Change sensor polling.
- Change dashboard behaviour.
- Change config schema casually.
- Mix old logger behaviour into v3 without naming the daily rotation choice.

## Acceptance checks

- Daily JSONL rotation is documented as the v3.1 decision.
- `OpenDecisions.md` is updated or the decision is closed there.
- Tests/docs cover the daily file naming and date rollover behaviour.
- `CHANGES.md` and `docs/v3.1/MigrationState.md` are updated.
