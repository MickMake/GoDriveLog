# Prompt — GoDriveLog v2.0.0 Fast Instrument Renderer Skeleton

Use this prompt in a fresh branch/chat to implement the first renderer pass properly.

## Objective

Implement v2.0.0 of the GoDriveLog display renderer change.

This pass establishes the new fast instrument dashboard as the primary display direction. It should not attempt to make the final visual design perfect. It should prove that the app can update a fixed 1920x480 Fyne dashboard quickly and reliably using direct object updates.

## Version

Target version: v2.0.0

Suggested branch:

```text
feature/v2-fast-instrument-renderer
```

Suggested PR title:

```text
v2.0.0 Add fast instrument dashboard renderer
```

## Required pre-checks

Before coding:

1. Confirm PR27 is merged into `main`.
2. Confirm PR28 is merged into `main`.
3. Pull/latest `main`.
4. Check for open/unmerged renderer/dashboard-related PRs or branches.
5. If an equivalent fast renderer branch/PR already exists, stop and report it.
6. Tag the current working old renderer baseline before changing the main display path.

Recommended tag:

```bash
git checkout main
git pull
git tag legacy-config-scene-dashboard
git push origin legacy-config-scene-dashboard
```

If the tag already exists, do not overwrite it. Report that it already exists and continue only if it points to the intended baseline.

## High-level goal

Replace the current config-driven dashboard as the main runtime display with a new fast fixed instrument dashboard.

The new path should be:

```text
sensor state -> instrument dashboard model -> direct Fyne canvas object updates
```

Not:

```text
sensor state -> generic decoders -> scene.Evaluate -> generic renderer.Update
```

## Hard guardrails

Do not:
- Change live OBD reader behaviour.
- Change OBD PID semantics.
- Modify `RaceDemoScenario` unless a genuine missing display value blocks this pass.
- Duplicate `RaceDemoScenario`.
- Build the new dashboard as a preference next to the old one.
- Preserve the old config-scene renderer as an equal runtime choice.
- Rewrite logging.
- Rewrite sensor polling.
- Rewrite the race demo provider.
- Refactor unrelated packages.
- Touch assets unless needed for the skeleton.
- Add external dependencies unless absolutely necessary.

Do:
- Keep the new renderer simple.
- Create the canvas object graph once.
- Update existing Fyne objects directly.
- Make the new instrument dashboard the normal app display path.
- Keep old code only as temporary legacy/reference if needed.
- Keep all changes focused on renderer/display wiring.

## Expected implementation shape

Create a new instrument dashboard package or file, for example:

```text
internal/ui/instrument_dashboard.go
```

or:

```text
internal/dashboard/instrument/dashboard.go
```

Use whatever location best fits the existing project, but avoid scattering the implementation across many packages.

The dashboard should expose something like:

```go
func NewInstrumentDashboard1920x480(store *sensors.StateStore) (*InstrumentDashboard, error)
func (d *InstrumentDashboard) CanvasObject() fyne.CanvasObject
func (d *InstrumentDashboard) Start(ctx context.Context, interval time.Duration)
func (d *InstrumentDashboard) Refresh()
```

Use fixed Fyne objects such as:
- `canvas.Text`
- `canvas.Rectangle`
- `canvas.Image` only where useful
- `container.NewWithoutLayout`

Initial skeleton can use native text. Do not block v2.0.0 on seven-segment art.

## Canvas

Target size:

```text
1920x480
```

The app window should open at 1920x480 for this new dashboard.

## Minimum displayed values

v2.0.0 must show at least:

- RPM
- speed
- throttle percentage
- oil temperature
- oil pressure
- gear
- warning level
- engine failed state
- requires reset state

It is acceptable for this pass to look plain. It is not acceptable for this pass to be static or slow.

## Data source

Use the existing `sensors.StateStore`.

Do not bypass the sensor polling path. The dashboard should read values from the store so race-demo and live OBD remain conceptually separate from display rendering.

Race demo should continue to launch with:

```bash
go run ./cmd/GoDriveLog --sensor-provider race-demo
```

If the config path is still required, use the existing config loading mechanism. Do not invent a separate app launcher unless necessary.

## Update cadence

Use a dashboard refresh interval suitable for live display:

```text
100ms preferred
250ms acceptable for first pass
```

Do not use `render_min_ms` throttling in the new instrument renderer.

## Old renderer handling

The old config-driven scene renderer should not be offered as a normal runtime preference.

Acceptable for v2.0.0:
- Leave old files in place temporarily.
- Stop routing normal app display through old renderer.
- Mention in PR that old renderer remains only as legacy/reference pending cleanup.

Not acceptable:
- Add `display.mode: old/new` as a normal user-facing preference.
- Present old and new renderers as equivalent supported paths.
- Make new renderer depend on scene.Evaluate.

## Verification

Run:

```bash
go test ./...
```

Manual verification:

```bash
go run ./cmd/GoDriveLog --config config.example.yaml --sensor-provider race-demo
```

Confirm:
- Window opens at 1920x480.
- RPM changes during the race demo.
- Speed changes during launch/cruise/braking.
- Throttle changes.
- Oil temp changes.
- Oil pressure drops to 0 after failure.
- Engine failed and reset states are visible.
- Updates are visually frequent enough to prove the new path is not frozen.
