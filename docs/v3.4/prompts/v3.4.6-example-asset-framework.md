# v3.4.6 - example asset generation framework

## Goal

Add the deterministic procedural asset generation framework for v3.4 example dashboards.

This slice establishes scripts, conventions, docs, and a minimal smoke-test output. It must not implement the final ornate timber, neon-grid, or steam-scrap dashboards yet.

## Required reading

1. `docs/v3.4/ReleasePlan.md`
2. `docs/v3.4/ImplementationState.md`
3. `docs/v3.4/GeneratedExamples.md`
4. `docs/v3.4/BaselineDashboardVerification.md` if present

## Pre-work checks

Before coding:

1. Confirm v3.4.5 is merged into `main`.
2. Confirm there are no blocking open v3.4 PRs.
3. Branch from latest `main`.
4. Use branch `v3.4.6-example-asset-framework`.

Stop and report if these checks fail.

## Scope

Add a local deterministic asset generation framework for example dashboards.

Expected work:

- Add a script entry point, for example `scripts/generate-example-assets.py`.
- Add helper modules only if they keep the code simpler, for example under `scripts/example_assets/`.
- Establish output conventions under `examples/<dashboard_name>/`.
- Establish dashboard example conventions so each example dashboard is self-contained under `examples/<dashboard_name>/`.
- Add a minimal smoke-test theme such as `framework-smoke` only if useful to verify the pipeline.
- Add or update docs explaining how to regenerate example assets.
- Update `CHANGES.md`.
- Update `docs/v3.4/ImplementationState.md`.

Keep this slice boring. Boring framework code is good framework code. Exciting framework code is usually a small fire wearing a lanyard.

## Reproducibility rules

- Use local procedural drawing only.
- Use stable seeds/config values for any noise or variation.
- Do not use remote image generation.
- Do not download stock art.
- Do not hand-edit generated PNGs as the source of truth.
- Commit scripts/config/docs needed to regenerate output.
- Generated outputs should be deterministic enough for code review.

## Runtime model rules

Do not alter the runtime gauge behaviour model.

Do not add runtime fields such as:

```yaml
style: ornate
```

Theme options inside generator code/config are fine. Runtime dashboard/gauge package YAML must continue to use the existing behaviour types:

```text
numeric
radial
odometer
indicator
bar
segmented
```

## Suggested framework capabilities

The framework should make later themed dashboards easy to implement:

- canvas creation and PNG writing;
- transparent layer support;
- basic shape drawing;
- deterministic noise/grain helpers;
- simple bevel/shadow helpers;
- reusable gauge layer helpers where practical;
- dashboard/package path conventions;
- clear CLI behaviour.

Keep dependencies minimal. Prefer the standard library plus already-accepted project tooling where practical. If a new dependency is needed, explain why.

## Smoke-test output

If adding smoke-test assets, keep them intentionally simple:

```text
examples/framework-smoke/dashboard.yaml
examples/framework-smoke/assets/
```

The smoke-test should prove that the generator can write PNGs and that the example path conventions work. It does not need to be pretty. It is the workshop jig, not the mantelpiece.

## Out of scope

- Ornate timber dashboard.
- Neon-grid dashboard.
- Steam-scrap dashboard.
- Renderer behaviour changes.
- New gauge types.
- Runtime style fields.
- Remote image generation.
- Downloaded artwork.
- Manual final-art editing.

## Validation

Run the generator.

Confirm:

- expected directories are created;
- expected PNG files are written;
- docs explain regeneration;
- existing tests still pass or known existing failures are documented;
- any existing dashboard/example validation command still passes if available.

## Done when

- The deterministic example asset generation framework exists.
- A regeneration command is documented.
- Optional smoke-test output is generated and committed if the repo convention supports generated examples.
- No final themed dashboard is implemented yet.
- `CHANGES.md` and `docs/v3.4/ImplementationState.md` are updated.
