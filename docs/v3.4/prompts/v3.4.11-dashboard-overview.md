# GoDriveLog v3.4.11 — Dashboard overview

You are working in the GoDriveLog repository.

Create the v3.4.11 dashboard overview slice.

This slice adds the bare dashboard command overview after v3.4.10 has established the dashboard CLI foundation, config discovery, `dashboard run`, and `dashboard validate`.

Do not add new renderer behaviour, gauge behaviour, harness behaviour, or generated example output in this slice.

## Required command shape

Implement:

```text
GoDriveLog dashboard [--config <config-file>]
```

The command prints a compact config overview. It should answer, "what does this config include?" without dumping the whole YAML.

## Config resolution

Use the config resolution behaviour already defined by v3.4.10:

- explicit positional config or `--config` loads exactly that file;
- otherwise search the current working directory, then `/etc/godrivelog` recursively;
- sort each directory's entries alphabetically before evaluating them;
- no vehicle ID means the first valid single-vehicle config wins, while the first valid multi-vehicle config errors and requires a vehicle ID;
- vehicle ID selection for the overview is not positional in this slice unless separately scoped later.

## Overview content

The overview must stay small and hierarchical.

Show:

- resolved config path;
- vehicles defined in the config;
- dashboards attached to each vehicle;
- gauges/widgets used by each dashboard;
- gauge/widget ID or name;
- gauge type where known;
- source of data;
- PID where the source is an OBD sensor.

Show the configured vehicle OBD source string exactly as defined in the vehicle config. Do not infer source type labels such as serial, Bluetooth, Wi-Fi/TCP, fake/test, or unknown.

Do not add a separate PID section.

Do not pretend virtual/static data has a PID.

## Failure behaviour

The overview command should be helpful without becoming a strict validation-only command.

If a dashboard or gauge package cannot be loaded, show a compact warning in the overview where practical. Keep hard validation failures in `dashboard validate` unless the missing data prevents any useful overview from being produced.
