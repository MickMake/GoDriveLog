# GoDriveLog v3.4.10 — Dashboard CLI

You are working in the GoDriveLog repository.

Create the v3.4.10 dashboard CLI slice.

This slice restructures the current flat `cmd/GoDriveLog` flags into a dashboard command tree. It is a CLI/runtime wiring slice, not a new renderer model and not a new gauge type slice.

## Required command shape

Implement these commands:

```text
GoDriveLog dashboard [--config <config-file>]

GoDriveLog dashboard run [vehicle-id]
  [--config <config-file>]
  [--renderer ebiten]

GoDriveLog dashboard harness [vehicle-id]
  [--config <config-file>]
  [--pattern sweep]
  [--interval 50ms]
  [--duration 60s]
  [--renderer ebiten]

GoDriveLog dashboard examples
  --output <directory>
  [--config <config-file>]
  [--vehicle <vehicle-id>]
  [--theme framework-smoke]
  [--force]

GoDriveLog dashboard validate [config-file]
GoDriveLog dashboard validate [--config <config-file>]
```

Do not add `dashboard preview` in this slice.

## Defaults and config resolution

`--config` is optional for commands that accept it.

When no config is supplied, resolve config files using the same directory search order as dashboard assets:

```text
config directory / vehicle ID
current working directory / vehicle ID
config directory
current working directory
/etc/godrivelog
/usr/local/etc/godrivelog
```

For config discovery, the starting config directory is the current working directory when there is no explicit config path. Use the same ordering concept as `internal/assets.DefaultSearchPaths` so config and asset lookup are predictable.

Do not default to `config.example.yaml`.

Candidate config filenames should be explicit and documented in the implementation. Suggested names:

```text
godrivelog.yaml
godrivelog.yml
dashboard.yaml
dashboard.yml
config.yaml
config.yml
```

If no config is found, return a useful error listing the searched locations.

`--renderer` is optional and defaults to `ebiten`. The only supported active renderer remains `ebiten`.

## Vehicle selection

`[vehicle-id]` is optional when the resolved config contains exactly one vehicle.

`[vehicle-id]` is required when the config contains multiple vehicles.

For `dashboard examples`, use `--vehicle <vehicle-id>` rather than a positional vehicle argument.

## Bare dashboard command

`GoDriveLog dashboard [--config <config-file>]` prints a compact config overview. It should be enough to answer, "what does this config include?" without dumping the whole YAML.

The overview must include vehicle data source information. For the selected or discovered vehicle, show whether the vehicle data source appears to be serial, Bluetooth, Wi-Fi/TCP, fake/test, or unknown. Derive this from the vehicle OBD-like address scheme where practical.

Example source interpretation:

```text
serial:///dev/ttyUSB0        -> serial
bluetooth://...              -> Bluetooth
tcp://127.0.0.1:35000       -> Wi-Fi/TCP or bench TCP
fake://...                   -> fake/test
```

The gauge listing should keep these columns or equivalent information:

```text
Gauge        Type        Data Source              PID
speed        numeric     sensor vehicle.speed     010D
rpm          radial      sensor engine.rpm        010C
coolant      segmented   sensor coolant_temp      0105
range        numeric     virtual fuel.range       -
warning      indicator   sensor engine.warning    -
```

Rules:

- Keep the output small and hierarchical.
- Show vehicles and their data source type.
- Show dashboards attached to each vehicle.
- Show gauges/widgets used by the dashboard.
- Keep the source of data visible.
- Show the PID where the source is an OBD sensor.
- Do not add a separate PID section.
- Do not pretend virtual/static data has a PID.

## Run command

`dashboard run` replaces the current normal runtime path.

It should use the same v3 selected-vehicle runtime currently used by the active Ebiten command path.

Do not change dashboard rendering semantics.

## Harness command

`dashboard harness` replaces the current `--harness` flag mode.

Keep existing pattern options for now. `sweep` remains the default.

For this slice, `sweep` should become gauge-aware synthetic input. It should drive each gauge in a way that matches the gauge behaviour instead of treating every gauge as the same numeric sweep.

Gauge-aware sweep behaviour:

| Gauge type | Sweep behaviour |
|---|---|
| `odometer` | Let `n = 0`; increment from `n - 20` to `n + 20` for 5 seconds, then `n + 20` to `n + 30` for 5 seconds. |
| `numeric` | Same as `odometer`. |
| `radial` | Keep the existing sweep style. |
| `indicator` | Flash on/off for 5 seconds with a 1s duty cycle, then flash on/off for 5 seconds with a 250ms duty cycle. |
| `bar` | Heartbeat pulse at 90 bpm. |
| `segmented` | Same input shape as `radial`. |

Do not over-expand `heartbeat` or `fixed` yet. If they look odd for some gauge types, leave that for a later scope decision.

## Examples command

`dashboard examples` replaces the ambiguity of a generic `generate` command.

`--output <directory>` is required.

If the output directory already exists and is non-empty, ask before overwriting. `--force` suppresses the question and allows clobbering.

Keep generated example output deterministic and local. Do not use remote image generation, stock art downloads, or hand-edited opaque PNGs as source of truth.

The first implementation may reuse or wrap the existing generated example asset machinery, but the public CLI shape should be `dashboard examples`.

## Validate command

`dashboard validate` validates the resolved config.

Allowed forms:

```text
GoDriveLog dashboard validate
GoDriveLog dashboard validate ./config.yaml
GoDriveLog dashboard validate --config ./config.yaml
```

Rules:

- If no config is supplied, use normal config discovery.
- If a positional config is supplied, validate that file.
- If `--config` is supplied, validate that file.
- If both positional config and `--config` are supplied, return an error.
